# regarde-ui

Regarde UI uses a three-layer architecture for building consistent, themeable components.

## Architecture

```
/base-ui/    → Base UI primitive re-exports (mergeProps, useRender, etc.)
/ui/  → Raw shadcn components (generated, pristine — DO NOT EDIT)
/atoms/      → Branded components (Regarde code, customizable)
```

| Layer       | Contains          | Editable                |
| ----------- | ----------------- | ----------------------- |
| `/base-ui/` | Base UI utilities | No — thin wrapper       |
| `/ui/`      | shadcn defaults   | No — regenerate only    |
| `/atoms/`   | Branded variants  | Yes — this is your code |

**Critical**: Never modify files in `/ui/`. These are regenerated by shadcn CLI.

## Core Rules

### Import Pattern

Always use the `Shad` prefix when importing from ui:

```typescript
// ✓ Good
import { Input as ShadInput } from "@/components/ui/input";

// ✗ Bad
import { Input } from "@/components/ui/input";
```

### Props Pattern

Always use `React.ComponentProps<typeof ShadComponent>`:

```typescript
// ✓ Good
export type InputProps = React.ComponentProps<typeof ShadInput> & {
  preSlot?: React.ReactNode
}

// ✗ Bad
export type InputProps = React.ComponentProps<"input"> & { ... }
```

### Ref Handling

Explicitly destructure `ref` in component params for React 19 **only when the component renders a DOM element**. Do NOT add explicit ref types — `React.ComponentProps` already includes ref:

```typescript
// ✓ Good — component renders DOM element
function Input({ ref, className, ...props }: InputProps) {
  return <ShadInput ref={ref} ... />
}

// ✓ Good — context provider, no DOM element, no ref needed
function DropdownMenu({ ...props }: DropdownMenuProps) {
  return <ShadDropdownMenu {...props} />
}

// ✗ Bad — explicit ref type is redundant
function Input({ ref, ...props }: InputProps & { ref?: React.RefObject<HTMLDivElement> }) {
  return <ShadInput ref={ref} ... />
}

// ✗ Bad — context provider doesn't accept ref
function DropdownMenu({ ref, ...props }: DropdownMenuProps) {
  return <ShadDropdownMenu ref={ref} {...props} /> // Error: ref doesn't exist
}
```

### File Naming

Atoms use **camelCase**:

```
// ✓ Good
packages/ui/src/components/atoms/input.tsx

// ✗ Bad
packages/ui/src/components/atoms/Input.tsx
```

### Class Merging

- Use `cn()` for conditional logic (`&&`, ternaries)
- Use `cn()` for **all** class merging in atom components (handles function classNames from Radix)
- Use `twMerge()` only for CVA variant definitions
- Use CVA only when implementing multiple style variants (size, intent, etc.)
- **Don't use CVA** if implementing variants via Tailwind data-attributes or arbitrary variants — remove unused CVA definitions

**Why `cn()` over `twMerge()` in atoms?**
Some shadcn components (DropdownMenu, Select, etc.) accept function classNames: `className?: string | ((state: State) => string)`. `twMerge()` only accepts strings and will throw TypeScript errors. `cn()` handles both strings and functions.

### Styling

Use semantic Tailwind classes that map to CSS variables:

```typescript
className={cn(
  "bg-input border-border text-foreground",
  "focus-visible:ring-ring",
  className
)}
```

### Documentation

Keep JSDoc minimal. Use `@example` for compound component usage patterns, omit verbose descriptions:

````typescript
/**
 * @example
 * ```tsx
 * <Card>
 *   <CardHeader>
 *     <CardTitle>Title</CardTitle>
 *   </CardHeader>
 * </Card>
 * ```
 */
const Card = ...
````

### Function Style

Use implicit return for simple wrapper components:

```typescript
// ✓ Good — implicit return with cn()
const Input = ({ ref, className, ...props }: InputProps) => (
  <ShadInput ref={ref} className={cn("bg-input", className)} {...props} />
)

// ✗ Bad — explicit return with braces for simple case
const Input = ({ ref, className, ...props }: InputProps) => {
  return <ShadInput ref={ref} className={cn("...", className)} {...props} />
}
```

## Migration from Dashboard to Atoms

Guidelines for migrating dashboard's local UI components to `/atoms/`:

### When to Create an Atom

**Create atom when:**

- Component is used in 2+ places
- Needs Regarde-specific styling (semantic tokens, domain features)
- Has reusable domain features (slots, loading states)

**Keep in dashboard when:**

- Feature-specific (only used in one feature)
- Complex business logic
- One-off styling overrides

### Migration Checklist

- [ ] Move file to `packages/ui/src/components/atoms/{name}.tsx`
- [ ] Update imports to use `Shad` prefix: `import { X as ShadX }`
- [ ] Ensure camelCase filename
- [ ] Add explicit ref destructuring: `function X({ ref, ... })`
- [ ] Use `cn()` for all class merging in atoms (handles function classNames from Radix)
- [ ] Export: `displayName` on each component; default export for root component; props types exported; for compound components, pick ONE pattern (see Pattern Standards)
- [ ] Update dashboard imports: `@regarde/ui/components/atoms/{name}`
- [ ] Delete local dashboard version
- [ ] Build: `pnpm --filter @regarde/ui build`

### Pattern Standards

Standardize on ONE pattern per component type:

| Component Type  | Pattern      | Examples                  |
| --------------- | ------------ | ------------------------- |
| Structure-heavy | Compound     | Card, Dialog, Sheet, Tabs |
| State-heavy     | CVA variants | Button, Badge             |
| Input elements  | Slot props   | Input (pre/post slots)    |

**Don't mix patterns** — pick one and be consistent.

**Compound pattern options** (choose ONE):

- **Option A**: Compound object only — `Card.Root`, `Card.Header` accessed via `Card.X`
- **Option B**: Individual exports only — `Card`, `CardHeader` as separate named exports
- **Option C**: Both compound AND individual — NOT recommended, creates two ways to do the same thing

## Domain Features to Add

When building atoms, consider these domain-specific additions:

- **Slots**: `preSlot`/`postSlot` for icons/actions (inputs, buttons)
- **Loading states**: `loading` prop with spinner overlay (buttons) — needed for external API calls
- **Layout utilities**: Use Tailwind classes directly (e.g., `w-full`)

Skip domain features that don't fit your component's purpose.
